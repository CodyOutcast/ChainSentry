# ChainSentry 后端训练与推理全流程说明

这份文档的目标是把下面几件事讲清楚：

1. 这个模型的目的是什么。
2. 训练时的真实输入输出是什么。
3. `/data` 中的数据是如何统一转换成训练样本的。
4. 模型的结构、参数、loss、optimizer、evaluation 是什么。
5. 后端推理时怎么跑。
6. demo 应该怎么设计输入，输出每个字段是什么意思，前端如何利用这些字段帮助用户分辨风险。


## 1. 这个模型的目的是什么

这个模型的定位不是“单独做最终裁决”的风控系统，而是整个后端 hybrid risk pipeline 中的一部分。

它的目标是：

- 从一笔交易构造一个本地交易图和一组标量特征。
- 对 3 类核心风险输出分数：
  - `approval`
  - `destination`
  - `simulation`
- 对整体严重度输出一个四分类分布：
  - `low`
  - `medium`
  - `high`
  - `critical`

然后，后端再把：

- 模型输出
- heuristic detector 输出
- heuristic simulation 输出

融合成最终给前端的用户可读风险报告。

所以最准确的描述是：

> 这是一个用于给交易模式打分的多头图模型，它服务于后端的 explainable risk report，而不是直接替代全部规则。


## 2. 整个系统的两条主链路

### 2.1 训练链路

训练时的流程是：

`/data` 原始公开数据  
-> dataset adaptor 映射成统一 `TransactionRequest` 壳样本  
-> `parse_transaction()`  
-> `simulation_engine.simulate()`  
-> `build_transaction_graph()`  
-> `extract_scalar_features()`  
-> `UnifiedTrainingSample`  
-> 模型训练  
-> 导出 `graph-model.pt` 和 `graph-model-metrics.json`

### 2.2 推理链路

推理时的流程是：

前端提交 `TransactionRequest`  
-> 后端 `parse_transaction()`  
-> 后端 `simulation_engine.simulate()`  
-> 构图和抽特征  
-> 图模型输出 risk scores  
-> detector 输出 heuristic findings  
-> 后端融合  
-> 返回 `AnalysisResponse`  
-> 前端渲染风险卡片


## 3. 外部输入、模型输入、模型输出、后端输出

### 3.1 外部 API 输入

前端或调用方真正提交给后端的是 `TransactionRequest`。

字段如下：

- `chain_id`
- `from_address`
- `to_address`
- `method_name`
- `calldata`
- `value_eth`
- `token_symbol`
- `token_amount`
- `approval_amount`
- `spender_address`
- `contract_name`
- `interaction_label`
- `notes`
- `simulation_profile`

这个结构定义在：

- `backend/app/models.py`

### 3.2 模型的真实输入

模型真正看到的不是原始 JSON，而是两部分：

- `TransactionGraph`
- `ScalarFeatureSet`

也就是说，原始交易会先被 parse 和 simulate，再被转换成：

1. 一张局部交易图
2. 一组数值/类别/布尔特征

### 3.3 模型输出

模型输出的是：

- `approval` 二分类风险分数
- `destination` 二分类风险分数
- `simulation` 二分类风险分数
- `severity` 四分类分布

此外训练中还有两个辅助 head：

- `address_malicious`
- `failure_aux`

它们主要用于训练，不直接暴露给前端。

### 3.4 后端最终输出

前端最终拿到的是 `AnalysisResponse`：

- `normalized_transaction`
- `overall_severity`
- `recommended_action`
- `summary`
- `findings`
- `simulation`

所以必须区分清楚：

- API 输入 != 模型输入
- 模型输出 != 最终前端输出


## 4. 训练时统一样本长什么样

统一训练样本结构叫 `UnifiedTrainingSample`。

它包含：

- `dataset_name`
- `sample_id`
- `normalized_transaction`
- `simulation`
- `graph`
- `features`
- `binary_targets`
- `binary_target_mask`
- `multiclass_targets`
- `multiclass_target_mask`
- `sample_weight`
- `metadata`

最重要的是两个 mask：

- `binary_target_mask`
- `multiclass_target_mask`

这表示：

- 不是每条样本都监督所有 head
- 只有 `mask=True` 的任务才参与 loss 计算

这使得来自不同来源的数据可以被统一到同一个多任务训练框架中。


## 5. 模型真正“看见了什么”

这一点非常重要，如果不讲清楚，很容易误以为模型读到了很多原始语义字段。

### 5.1 图输入里实际被编码的内容

图在构建时会创建：

- transaction node
- address node
- contract node
- token node
- effect node

以及边：

- `initiates`
- `targets`
- `approves`
- `requests_allowance_for`
- `transfers_value_to`
- `transfers_token_to`
- `routes_to`
- `grants_operator_to`
- `grants_privilege_to`
- `triggers_effect`

但是当前 `vectorization` 真正编码给模型的只有：

- node type
- edge type
- anchor node index

也就是说，当前模型**没有直接编码**下面这些原始值：

- 地址字符串本身
- `contract_name`
- `interaction_label`
- `notes`
- token symbol 的具体 identity
- node/edge attribute 中的文本内容

这些字段要么只参与样本构造和 summary，要么只保存在 metadata 中。

### 5.2 标量特征

当前 artifact 中的真实特征字段如下。

数值特征：

- `approval_amount`
- `chain_id`
- `graph_edge_count`
- `graph_node_count`
- `simulation_effect_count`
- `token_amount`
- `value_eth`

布尔特征：

- `has_selector`
- `has_spender`
- `has_token`
- `simulation_triggered`

类别特征：

- `method_name`
- `selector`
- `simulation_profile`
- `transaction_kind`

这意味着当前模型主要依赖的是：

- 交易类型
- 方法名/selector
- 金额
- 是否有 spender / token / simulation
- 图的结构形状

而不是地址 identity 本身。


## 6. `/data` 中哪些数据真正用于默认训练

当前默认训练只使用了 5 个 adaptor：

- `forta`
- `eth-labels`
- `etherscamdb`
- `ptxphish`
- `raven`

没有接入默认训练主路径的数据包括：

- `forta-malicious-smart-contract-dataset`
- `ethereum_fraud_dataset_by_activity`
- `ethereum-etl`

所以不能说“`/data` 里的全部数据都参与了这次训练”。


## 7. 每个数据集是怎么统一转换的

下面每个数据集都给一个真实样例，展示：

- 原始字段
- 统一后的训练样本
- 哪些字段真正参与训练
- 哪些字段只是 metadata 或展示辅助


## 7.1 Forta

### 原始记录示例

```json
{
  "address": "0x000000000532b45f47779fce440748893b257865",
  "source": "forta-etherscan-malicious",
  "label": "Fake_Phishing3901",
  "notes": ""
}
```

### Forta 如何用

Forta 会被投影成两类壳样本：

1. `destination` 正样本
2. `approval` 正样本

### A. destination 壳样本

统一后的关键字段：

- `to_address = 原始 address`
- `method_name = claimRewards`
- `contract_name = Forta Flagged Destination`
- `interaction_label = Claim campaign rewards`
- `notes = 原始 notes 或 label`
- `simulation_profile = none`

监督标签：

- `destination = 1`
- `address_malicious = 1`

不监督：

- `approval`
- `simulation`
- `severity`

### B. approval 壳样本

统一后的关键字段：

- `to_address = 原始 address`
- `spender_address = 原始 address`
- `method_name = approve`
- `token_symbol = USDC/USDT/WETH/DAI` 之一
- `approval_amount = 1000000000`
- `simulation_profile = allowance_drain`

监督标签：

- `approval = 1`
- `address_malicious = 1`

不监督：

- `destination`
- `simulation`
- `severity`

### 原始字段参与情况

- `address`：参与，成为 `to_address` / `spender_address`
- `source`：不进模型输入，只进 `metadata`
- `label`：不进模型主输入，只进 `metadata`，也可能用于 `notes`
- `notes`：会进 request，但当前模型不直接编码 `notes` 文本


## 7.2 eth-labels

### 原始记录示例

```json
{
  "address": "0x0000000000001ff3684f28c67538d4d072c22734",
  "label": "0x-protocol",
  "name_tag": "0x: Allowance Holder"
}
```

### eth-labels 如何用

这个数据集当前被当作**低风险负样本来源**。

代码里只保留 allowlist 中的 benign label，例如：

- `aave`
- `sushiswap`
- `yearn`
- `0x-protocol`

然后每个地址被投影成 3 种低风险壳样本：

1. `destination` 负样本
2. `approval` 负样本
3. `simulation` 负样本

### A. destination 负样本

- `to_address = 原始 address`
- `method_name = contractCall`
- `contract_name = name_tag`
- `simulation_profile = none`

监督：

- `destination = 0`
- `address_malicious = 0`
- `severity = low`

### B. approval 负样本

- `method_name = approve`
- `spender_address = 原始 address`
- `approval_amount = 250 + 微小扰动`

监督：

- `approval = 0`
- `address_malicious = 0`
- `severity = low`

### C. simulation 负样本

- `method_name = swapExactTokensForTokens`
- `spender_address = 原始 address`

监督：

- `simulation = 0`
- `address_malicious = 0`
- `severity = low`

### 原始字段参与情况

- `address`：参与，成为 `to_address` / `spender_address`
- `label`：先用于 allowlist 过滤，不直接进入模型主输入
- `name_tag`：进入 `contract_name`，但模型不直接编码 contract name 文本

### 原始记录示例

```json
{
  "address": "0x0020731604c882cf7bf8c444be97d17b19ea4316",
  "category": "Scamming",
  "subcategory": "Trust-Trading",
  "name": "decentralized-launch.com",
```

### 映射结果

- `address` -> `to_address`
- `subcategory` 或 `category` -> `contract_name`

- `destination = 1`

- `address`：参与
- `category` / `subcategory`：只通过 `contract_name` 间接出现，但当前模型不直接编码这段文本
- `name` / `url`：只进 `notes` 或 metadata，不直接进模型


## 7.5 Raven

### 原始记录示例

Raven 的原始字段包括：

- `hash`
- `from_address`
- `to_address`
- `tx_input`
- `value`
- `failure_reason`
- `failure_message`
- `failure_invariant`

### 映射结果

- `hash` -> `sample_id`
- `from_address` -> `from_address`
- `to_address` -> `to_address`
- `tx_input` -> `calldata`
- `value` -> `value_eth`
- `failure_reason/message/invariant` -> `notes` 和 metadata

监督：

- 只监督 `failure_aux = 1`

不监督：

- `approval`
- `destination`
- `simulation`
- `severity`

### 原始字段参与情况

- `from_address` / `to_address` / `tx_input` / `value`：参与
- `failure_reason/message/invariant`：不直接进入模型主特征，只作为上下文和 metadata

### Raven 在训练中的作用

Raven 主要是辅助任务数据，用来训练失败/异常交易相关的辅助 head，不是主三类风险监督数据。


## 8. 训练时的 split 和 evaluation

### 8.1 普通数据集 split

默认 adaptor 使用固定随机种子做：

- train = 80%
- val = 10%
- test = 10%

### 8.2 Raven 的 split

Raven 比较特殊：

- `train/val` 使用 `finetuning` parquet
- `test` 使用单独的 `evaluation` parquet

所以 Raven 确实存在单独 evaluation 数据。

### 8.3 当前 standard 训练的实际规模

当前 artifact 对应的 `standard` 配置下：

- train: `4650`
- val: `1120`
- test: `1120`

test 的数据集构成：

- `forta`: 250
- `eth-labels`: 540
- `etherscamdb`: 120
- `ptxphish`: 30
- `raven`: 180

### 8.4 当前 metrics 的解读

当前 `graph-model-metrics.json` 中：

- approval / destination / simulation / severity 的很多指标接近或等于 `1.0`
- 但是 `severity` 的 `medium` 类在 val/test 中 `support = 0`

所以最准确的表述应该是：

> 训练流程跑通了，artifact 产出了，evaluation 也有；但这个 evaluation 更像统一壳样本上的内部评测，不能直接等价为对真实链上分布的强泛化证明。


## 9. 模型结构

### 9.1 当前 artifact 的真实配置

当前加载出来的 `RelationAwareGraphModelConfig` 是：

- `hidden_dim = 64`
- `relation_layers = 2`
- `categorical_embedding_dim = 8`
- `feature_hidden_dim = 48`
- `head_hidden_dim = 64`
- `dropout = 0.15`

总参数量：

- `179,697`

可训练参数量：

- `179,697`

### 9.2 架构概括

模型由 4 个部分组成：

1. node type embedding
2. relation-aware graph message passing
3. scalar feature encoder
4. 多头预测 head

### 9.3 图部分

图部分做法是：

- 对 node type 做 embedding
- 做 2 层 relation-aware passing
- 每一层都有：
  - 一个 self linear
  - 多个 relation-specific linear

最后聚合出：

- `anchor_state`
- `graph_mean`
- `graph_max`

### 9.4 标量特征部分

标量特征输入由三部分拼起来：

- numeric
- boolean
- categorical embedding

然后通过：

- `Linear -> ReLU -> Dropout`

得到一个 feature hidden vector。

### 9.5 最终拼接维度

最终 head 的输入维度是：

- `anchor_state` 64
- `graph_mean` 64
- `graph_max` 64
- `encoded_features` 48

总计：

- `240`

### 9.6 输出头

当前有 6 个 head：

binary heads：

- `approval`
- `destination`
- `simulation`
- `address_malicious`
- `failure_aux`

multiclass head：

- `severity`

每个 binary head 的结构是：

- `Linear(240, 64)`
- `ReLU`
- `Dropout`
- `Linear(64, 1)`

severity head 的结构是：

- `Linear(240, 64)`
- `ReLU`
- `Dropout`
- `Linear(64, 4)`

## 11. loss 和 optimizer

### 11.1 optimizer

使用：

- `AdamW`

参数：

- `lr = 3e-4`
- `weight_decay = 1e-4`

### 11.2 binary head loss

binary task 使用：

- `BCEWithLogitsLoss`

并且会根据 train set 上该 head 的正负样本比例，动态计算：

- `pos_weight = negatives / positives`

### 11.3 severity loss

severity 使用：

- `CrossEntropyLoss`

并根据类别频次构造 class weight。

### 11.4 各任务 loss 权重

binary：

- `approval = 1.0`
- `destination = 1.0`
- `simulation = 1.0`
- `address_malicious = 0.5`
- `failure_aux = 0.35`

multiclass：

- `severity = 0.75`

### 11.5 threshold

主三个 binary head 会在 val 集上搜索阈值，搜索空间为：

- `0.3`
- `0.35`
- `0.4`
- `0.45`
- `0.5`
- `0.55`
- `0.6`
- `0.65`

当前 artifact 中最终阈值为：

- `approval = 0.3`
- `destination = 0.3`
- `simulation = 0.3`
- `address_malicious = 0.5`
- `failure_aux = 0.5`


## 12. 后端推理是怎么运行的

### 12.1 第一步：接收 API 输入

前端通过 `POST /api/v1/analyze` 提交 `TransactionRequest`。

### 12.2 第二步：parse

`parse_transaction()` 做的事情：

- 从 `calldata` 取 selector
- 推断 method name
- 推断 `transaction_kind`
- 生成用户可读 `summary`

比如：

- `approve` -> `approval`
- `swapExactTokensForTokens` -> `swap`
- `grantRole` -> `privilege`

### 12.3 第三步：simulation

当前 simulation 是 heuristic engine，不是 Foundry fork。

它会根据：

- `transaction_kind`
- `method_name`
- `simulation_profile`

生成 effect，例如：

- `allowance_grant`
- `operator_control`
- `privilege_grant`
- `unexpected_outflow`

### 12.4 第四步：模型推理

后端做：

- 构图
- 抽特征
- 调 `LoadedGraphRiskModel.predict()`

得到：

- 三个 risk score
- 一个 severity 分布

### 12.5 第五步：规则检测

同时运行 heuristic detector：

- approval detector
- flagged destination detector
- simulation effect detector

### 12.6 第六步：融合

后端会把：

- heuristic findings
- model score
- model evidence

融合成最终 findings。

如果 heuristic finding 已存在，则把模型 evidence 追加进去；  
如果 heuristic finding 不存在，但模型 score 超过阈值，后端可能补一个 generic model finding。

## 14. 后端返回的每个字段是什么意思

### 14.1 `normalized_transaction`

这是后端解析后的交易语义，不是原始请求的机械复制。

字段含义：

- `transaction_kind`：后端判断的交易类型
- `method_name`：真实使用的方法名
- `selector`：如果有 calldata，则给出 selector
- `summary`：给用户读的摘要句

### 14.2 `overall_severity`

整笔交易的总风险等级：

- `low`
- `medium`
- `high`
- `critical`

### 14.3 `recommended_action`

给用户的动作建议：

- `proceed`
- `inspect_further`
- `reject`

### 14.4 `summary`

顶部一句总结，例如：

- `ChainSentry found 4 risk signals. Highest severity: critical.`
- `No configured ChainSentry risk signals were detected for this transaction.`

### 14.5 `findings`

每条 finding 代表一个具体风险点。

每个 finding 包含：

- `id`：规则或模板 ID
- `category`：`approval` / `destination` / `simulation`
- `severity`
- `expected_action`
- `risk_reason`
- `possible_impact`
- `recommended_action`
- `evidence`

### 14.6 `simulation`

simulation 部分告诉前端：

- 当前 simulation engine 是什么
- 采用了什么 profile
- 是否触发了 effect
- effect 的文本说明是什么


## 15. 前端是怎么利用这些输出帮助用户分辨风险的

前端流程：

1. 用户在 `TransactionForm` 填写或加载 demo scenario
2. 前端把 `TransactionDraft` 转成 `TransactionRequest`
3. 调用 `/api/v1/analyze`
4. 收到 `AnalysisResponse`
5. `RiskReport` 渲染结果

### 15.1 顶部层

前端首先显示：

- `analysis.summary`
- `overall_severity`
- `recommended_action`

这让用户第一眼就知道：

- 风险高不高
- 应该继续、再看，还是拒绝

### 15.2 “Expected action” 层

前端接着显示：

- `normalized_transaction.summary`

它的作用是先回答：

> 你现在实际上是在签什么？

### 15.3 finding 卡片层

每张 finding 卡片展示：

- `Expected action`
- `Risk reason`
- `Possible impact`
- `Recommended action`
- `evidence`

这对应用户签名前最关心的四个问题：

1. 我在做什么？
2. 为什么危险？
3. 会造成什么后果？
4. 我应该怎么办？

### 15.4 simulation 与 transaction details 层

最后前端还补充显示：

- simulation 结果
- transaction details

用于支持更细的解释，但不会取代顶部结论。


## 16. 一句话总结整个后端训练和推理

如果要用一句最准确的话概括：

> ChainSentry 的训练流程，是把多个公开数据集先投影成统一的交易壳样本，再把这些样本转换成局部交易图和标量特征，训练一个多头图模型去预测 approval、destination、simulation 和 severity；推理时，前端提交一笔交易，后端先解析和 heuristic simulation，再用图模型打分，并与规则结果融合，最终返回用户能看懂的风险卡片。


## 17. 当前版本最需要诚实说明的限制

为了避免答辩时被追问时说乱，下面这几点应该直接承认：

- 当前 simulation 是 heuristic，不是 Foundry fork simulation。
- 当前模型不直接编码地址 identity、合约名文本、notes 文本。
- 当前 evaluation 存在，但主要是在统一壳样本上进行。
- 当前系统最终输出是 `model + rules + post-processing` 的融合结果，不是“模型单独判定全部风险”。
