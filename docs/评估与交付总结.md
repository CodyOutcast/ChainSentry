# ChainSentry 评估与交付总结

## 1. 当前交付状态

当前仓库已经达到课程原型的完整闭环状态，包含：

- 前端：可连接钱包、加载 demo 场景、提交交易分析请求
- 后端：可解析交易、运行启发式检测、运行本地 simulation、调用图模型推理
- 模型：可复现训练、artifact 导出、metrics 导出、后端自动加载与 fallback
- 文档：接口文档、Student 2 说明、用户中文使用说明、交付总结

这意味着项目已经不再只是设计草案，而是可以实际演示的 explainable prototype。

## 2. 对老师反馈的回应

老师在 `docs/chainsentry.txt` 中指出两个核心问题：

1. 原始 proposal 的技术野心过大
2. 项目应更关注用户如何理解风险，而不是继续堆技术模块

当前实现已经对第一个问题做出明确收缩：

- 不做 full static analysis
- 不做 web-scale 动态图系统
- 不接生产级外部风控 API
- 不做多链生产部署
- 只聚焦少量典型、容易解释的签名前风险

当前实现对第二个问题的回应方式是：

- 风险输出统一为 `Expected action / Risk reason / Possible impact / Recommended action`
- 前端展示尽量保持短、清楚、可行动
- 用 4 个典型场景做 scenario-based review，而不是扩成复杂平台

当前不足也需要诚实说明：

- 目前完成的是 scenario-based evaluation 与 usability-oriented wording review
- 还没有正式的大规模真实用户实验

## 3. 当前收缩后的项目范围

当前系统只处理三类风险：

- 授权风险：大额或无限 token approval
- 目标风险：与本地 flagged-contract 数据集中的地址交互
- 隐藏后果风险：simulation 暗示的异常 outflow、operator control、privilege escalation

当前前端内置 4 个演示场景：

- Unlimited approval to flagged contract
- Collection-wide operator approval
- Swap route with hidden outflow
- Clean transfer baseline

这和老师建议的“宁可小而扎实，也不要做成泛化的大平台”一致。

## 4. 当前模型与评测结果

默认训练命令：

  --artifact-path backend/app/ml/artifacts/graph-model.pt \
  --metrics-path backend/app/ml/artifacts/graph-model-metrics.json

## 5. 解释层与可用性检查

当前系统的解释层已经满足“用户先看懂再决策”的最低设计要求：

- 每个 finding 都明确说明交易在做什么
- 每个 finding 都说明为什么触发风险
- 每个 finding 都给出可能后果
- 每个 finding 都给出下一步建议
- 顶部统一给出整体 severity 与推荐动作

当前更接近“轻量可用性审查”而不是正式用户研究，已完成的可说明点包括：

- 风险文案保持短句
- 风险等级区分为 low / medium / high / critical
- 推荐动作统一为 proceed / inspect further / reject
- 通过 4 个典型场景检查 explanation 是否过长、是否能落到用户决策

## 6. 剩余边界与诚实说明

项目已经完成课程原型交付，但仍有边界：

- 数据集是 synthetic + pseudo-label，不是真实标注链上数据
- simulation 仍是 heuristic engine，不是 Foundry trace 或生产级沙箱
- flagged contract 数据来自本地 demo 数据，不是实时威胁情报源
- 尚未完成正式 live user study

## 7. 课程答辩时可以怎样概括

可以用下面这段话：

> 我们根据老师的反馈，主动收缩了项目范围，没有继续堆更多技术模块，而是把系统聚焦到少量典型签名前风险，并把输出统一压缩成用户能快速理解的风险卡片。在这个收缩范围内，我们已经完成了前后端闭环、图模型训练与推理接入、场景化评测和中文使用文档。当前版本适合课程原型展示，不宣称是生产级风控系统。
